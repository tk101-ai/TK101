3. 아키텍처 결정 — Clean Architecture 채택 근거

3.1 3가지 옵션 비교

기준A: MinimalB: Clean Architecture ★C: Pragmatic
구조평면 폴더4-layer 엄격 분리Feature-based 모듈
새 파일 수~25~60~40
복잡도낮음높음중간
유지보수성낮음 (3년 시점 부담)높음 (3년 이후에도 견고)높음
초기 세팅 시간1일3일2일
회계 모듈 정확도낮음 (DB 의존 테스트)높음 (Domain 순수 테스트)중간
멀티 AI 전환 용이성낮음높음 (Adapter 패턴)중간

3.2 B 선택 결정 이유

핵심 판단: 4/21 미팅 시연 제약을 배제할 수 있다면(→ 5월 중순까지 회계 완성 가능), 장기 관점에서 B가 명확히 우위. Claude Code로 유지보수하는 전제에서도 초기 세팅 3일 투자가 3년 운영 혜택으로 상환됨.
  1. 회계 로직 정확도 담보 — 은행 파싱/세금계산서 매칭은 돈을 다루는 로직. Domain 레이어 순수 함수로 DB 없이 단위 테스트 가능 → 이사님께 "매칭 정확도 검증됨" 당당히 보고
  2. 멀티 AI 전환 대비 — 3단계 OpenAI/Gemini 추가 계획. Infrastructure 레이어 Adapter 패턴으로 Claude → 타 AI 교체 시 다른 코드 수정 최소화
  3. 5개 팀 모듈 확장성 — 마케팅1/2, 디자인, 신사업, 회계 각 모듈이 bounded context. 패턴이 확립되면 Claude Code가 새 모듈을 빠르게 복제 가능
  4. END USER 비용 무관 — 레이어 증가가 API 호출 경로를 늘리지 않음. 직원 토큰 비용은 아키텍처와 독립
  5. 테스트 커버리지 80%+ 달성 용이 — Domain 레이어는 외부 의존성 0 → TDD가 매우 쾌적

3.3 4 레이어 구조

┌──────────────────────────────────────────────────────────────┐

│    FastAPI routers, Pydantic schemas, middleware              │

4. 도구 전략 — bkit + ECC 병행

4.1 역할 분담

  • bkit = 프로세스 매니저 (PDCA 흐름, 기획, QA 단계, 배포 관리)
  • ECC = 기술 전문가 (언어별 코드 리뷰, 보안, TDD, 빌드 해결)
  • 중첩 금지: 같은 작업을 두 도구로 동시에 하지 않음

4.2 표준 13단계 플로우 (기능 1개 개발 시)

#단계명령어주도
1기획/bkit:pdca plan <feature>bkit
2설계 (아키텍처 옵션 선택)/bkit:pdca design <feature>bkit
3UI 목업 (선택)/bkit:phase-3-mockupbkit
4구현 계획 세부화/plan <feature>ECC
5테스트 먼저 작성 (L0~L3)/tdd <feature>ECC
6코드 작성 (자동 리뷰)저장 시 python-reviewer/typescript-reviewer 자동ECC
7구현 진행 추적/bkit:pdca do <feature> --scope Xbkit
8QA 실행/bkit:qa-phase <feature>bkit + ECC e2e-runner
9갭 분석 (90%+ 목표)/bkit:pdca analyze <feature>bkit
10자동 개선 (필요시)/bkit:pdca iterate <feature>bkit
11완료 보고서/bkit:pdca report <feature>bkit
12배포git push → GitHub Actions 자동-
13아카이브/bkit:pdca archive <feature> --summarybkit

5.4 희석 방지 원칙

  • 구체적 규칙 > 추상적 규칙 (예: "파일 ≤ 400줄" 은 도구로 검증 가능)
  • 중복 제거 (같은 정보를 두 곳에 두지 않음)
  • 동적 vs 정적 분리 (현재 진행 상태는 별도 파일)
  • 명령형 표현 (권장형 "고려하세요" 금지)
